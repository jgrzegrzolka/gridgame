# Daily release incidents

Time-ordered journal of failures that affect the daily-puzzle release path — `func-yetanotherquiz-release` → `live.json`. Newest incident at the top. One section per night. Each section records what we saw, what we ruled in or out, what we did, and what's still open.

`infra/operations.md` is the *runbook* — durable "if you see X, do Y". This file is the *journal* — the messy story behind why a runbook entry exists, and a holding place for symptoms that don't yet have a clean fix.

## 2026-06-17 (Warsaw midnight) — Function host wedged after a Bicep redeploy; today's N=12 not promoted

### Symptom
At ~00:11 Warsaw (22:11 UTC on 2026-06-16, ~6 min after the expected fire) `/daily/` showed yesterday's puzzle (N=11). `backlog[0]` was N=12 (Europe + animal) and had never been promoted.

### Diagnosis (in the order the evidence landed)

1. **Catalog state.** `live.json` Last-Modified = **14:37:43 UTC on 2026-06-16** — hours earlier than the 22:05 UTC fire would produce. Blob version history shows three writes today, all in the early afternoon:
   - `12:51:00 UTC` — N=11 manual promote (catch-up from the previous night's miss — see the 2026-06-16 section below).
   - `14:33:16 UTC` — size jumped from 4559 → 4696 (≈ +137 B). Consistent with the Function promoting N=12 to `live`.
   - `14:37:43 UTC` — back to 4559 B (N=11 only).
   
   The `14:33 → 14:37` snapshot+restore pair is the signature of the manual smoke-test pattern in `infra/README.md`. **So the Function was healthy as of 14:33 UTC today.**

2. **Function App state.** `az functionapp show` reports `state: Running`, `linuxFxVersion: Node|22`. But `az functionapp function list` returns **empty** — no `releaseDaily` registered → timer never fires.

3. **Activity log shows a redeploy by Jan between 17:39–17:47 UTC today:**
   - `Update hosting plan` (17:39:26)
   - `Update website` (17:39:52)
   - `Update web sites config` (17:41:43–45)
   - `Sync Web Apps Function Triggers` (17:41:45–47)
   - `Restart Web App` (17:47:01)
   
   Same shape as the runbook in `operations.md` ("Re-deploying `funcapp-release-daily.bicep` silently breaks the Function"). **But unlike that runbook's case, `WEBSITE_RUN_FROM_PACKAGE` is NOT stripped** — it points at a fresh zip uploaded at ~17:41 UTC. The host nevertheless refuses to register the function.

4. **Host runtime.** `GET /admin/host/status` returns **HTTP 503 — "ServiceUnavailable from host runtime"**. The runtime is wedged below the admin surface. `POST .../syncfunctiontriggers` also fails with the same `ServiceUnavailable from host runtime` error.

5. **App Insights.** Workspace `ai-yetanotherquiz-release` has **zero rows** across `traces`, `exceptions`, and `requests` for the full retention window queried (12 h). The worker isn't getting far enough to emit telemetry — startup is failing before `APPLICATIONINSIGHTS_CONNECTION_STRING` is honoured.

### Recovery attempts in this session (all unsuccessful)

- Rebuilt with `node scripts/build-release-fn.mjs` → fresh `infra/release-fn/dist/` (`host.json`, `index.mjs`, `package.json`, `node_modules/`).
- Re-zipped with Python (forward-slash entries, per the runbook in `infra/README.md`).
- `az functionapp deployment source config-zip` returned `Bad Request` at the CLI but **did** update `WEBSITE_RUN_FROM_PACKAGE` to a fresh `20260617001934-a288aaf4-…` URL — upload reached storage, post-upload sync failed.
- `az functionapp stop` → `az functionapp start`. Host returned 503 for 5+ minutes after restart.
- Polled `/admin/host/status` six times over ~1.5 min after restart — always 503.

The user-facing fix (manually promoting `backlog[0]` → end of `live.json` via the `daily-puzzle-author` skill) is being tracked as a separate immediate action — it doesn't touch the Function.

### Root cause hypothesis (unconfirmed — investigate next session)

Something the 17:39 UTC Bicep redeploy changed is preventing the worker from booting at all. Candidates to check tomorrow, in priority order:

1. **Diff `infra/funcapp-release-daily.bicep` and the rest of the working tree** against the commit that produced the 14:33 UTC healthy run. Whatever the Bicep change was, that's the highest-signal artefact.
2. **`AzureWebJobsStorage` reachability.** The connection string in app settings points at `stfuncyetanotherquiz` with the embedded account key. Verify the key is still valid (key rotation? regenerated by Bicep?). The runtime needs this storage account to come up at all — broken storage = silent boot failure, no telemetry.
3. **`AzureWebJobsFeatureFlags=EnableWorkerIndexing`.** Required by the v4 programmatic model. Confirmed present in app settings as of this incident, but worth re-verifying after any Bicep round-trip.
4. **Linux Consumption regional health (West Europe).** Check `resourcehealth` for the Function App resource — a regional content-distribution or worker-pool issue would manifest exactly like this.
5. **Function App's managed identity ↔ `styetanotherquiz` role assignment.** Less likely to manifest as "host won't start" (boot precedes blob access), but worth a sanity check if other paths are clean.

## 2026-06-16 (Warsaw midnight) — Reconstructed: midnight Function didn't fire; N=11 promoted manually at 12:51 UTC

### What we know — and what we don't

The visible blob version history for `live.json` starts at **12:51:00 UTC on 2026-06-16** (4559 B, the N=11 catalog). Any version that would have been written by the 22:05 UTC fire on 2026-06-15 is not in the listing — either it was never written or it's been trimmed.

App Insights `ai-yetanotherquiz-release` is silent for the full window we can query — same as today's incident. That's evidence the worker had already been wedged before today's Bicep attempt; today's redeploy was Jan trying to fix the prior night's miss, not introducing a new failure on a previously healthy host.

That theory has one inconsistency: the 14:33 UTC smoke-test promoted N=12 cleanly. So at 14:33 UTC the host was clearly alive enough to run a function. Either:

- Whatever Jan did between 12:51 UTC (the manual N=11 promote) and 14:33 UTC (the smoke test) revived the host temporarily, and the 17:39 UTC Bicep redeploy then broke it again, **or**
- The host had been intermittent — fine for the 14:33 UTC manual fire, silent at the 2026-06-15→16 midnight fire, and silent again from 17:47 UTC onward.

Carry into tomorrow's investigation:

- When poking at `funcapp-release-daily.bicep` and the runtime, also check whether the App Insights workspace ever recorded a successful invocation in the last 7 days. If AI has been silent for longer than today, the underlying breakage started before today's redeploy attempt.
- If the version-history truncation is from a blob lifecycle policy, get that policy set wider so the next incident has more data to work from. Storage account is `styetanotherquiz` / container `catalog`.

### What was done in the moment

Jan ran a manual promote of N=11 at **12:51 UTC on 2026-06-16** — most likely via the `daily-puzzle-author` skill (`authoring/pull.mjs` → edit `.catalog/live.json` → `authoring/push.mjs`). Player-facing recovery latency: ~13 hours past Warsaw midnight, mitigated by the manual promote.
